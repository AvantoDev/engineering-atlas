A User Story is the hinge between the two zones — strategy hands it down, execution builds it up.

Level 1 · Strategy

Epic — the strategic initiative

✅What it is. A large capability the Strata program wants to deliver — broad on purpose, spanning many features and releases.
👥Who owns it. Business, Architecture, and the SMEs — defined and refined in business / architecture grooming.
🔗How it connects. Contains many User Stories. It is not tied to a release — Epics stay open for months while their stories ship one by one.
Mental model — “What strategic capability are we building?”
Examples
Multi-currency support across the platformEnd-to-end document automation (OCR → classification → export)Self-service tenant onboarding
Epic · owned by Business + Architecture + SMEs
Level 2 · Strategy

User Story — the versionable feature

📦The release unit. A release ships one or more User Stories — never partial Tasks.
👥Who owns it. Product (scope, acceptance) and Architecture (technical coherence). Engineering is consulted, not the owner.
🔗How it connects. One Epic → several Stories. One Story → several Tasks.
Mental model — “Which slice of the Epic ships next?”
🌿 Branching rule (mandatory)
Create a dedicated branch from production, named with the User Story ID.branch name = <US-ID>e.g. ST-1234 — all Tasks of that Story merge into this branch; the branch is what gets promoted.
User Story · owned by Product + Architecture · the release unit
Level 3 · Execution

Task — the implementation unit

✅What it is. A piece of implementation work for a Story. One Story normally has several Tasks across disciplines: backend, frontend, devops, research, spike, data, infra, QA.
👥Who owns it. The Engineering Lead decomposes the Story into Tasks; the developer / squad picks them up.
👯Release behavior. Tasks never ship alone. If a Task misses the cutoff, the whole Story typically slips.
Mental model — “What does engineering have to build to deliver this feature?”
🔗 Maps to Beadbox
Jira Task = Initech Epic
Pick up a Task → create the matching Initech Epic in Beadbox → decompose it locally into Beads.
Task · owned by Engineering · = Initech Epic in Beadbox
Level 4 · Execution

Sub-task — the atomic work item

✅What it is. The smallest tracked unit — a commit-sized change, a test, a config update, a small refactor, or a focused investigation.
👥Who owns it. The developer actively working the Task.
🔗How it connects. One Task → several Sub-tasks. A Sub-task ships implicitly when its parent Task ships.
Mental model — “Which concrete step am I working on right now?”
🔗 Maps to Beadbox
Jira Sub-task = Bead
For every Bead worked locally, there is a matching Sub-task in Jira. The two are a 1:1 mirror.
Sub-task · owned by Developer · = Bead in Beadbox
Jira ↔ Beadbox

Execution levels mirror your local stack

Level 3
Task ⟷ Initech Epic
The Jira Task is the contract; the Initech Epic is where you actually break it down and work it.
Level 4
Sub-task ⟷ Bead
Each Bead you ship has a twin Sub-task in Jira. Move one, move the other.

Why it matters — Jira is the long-term source of truth and the shared line of sight for Product, Architecture and Leadership. Beadbox is the local execution lens.

🪬 Non-negotiable

The mirror discipline

✅ What developers must do

→Bead moves to In Progress → transition the matching Sub-task in Jira.
→Bead closed → close the Sub-task with a short summary comment.
→Initech Epic done → summary comment on the Jira Task with PR links & key decisions.
→Blockers, scope changes & dependencies surface as comments on the Jira side.

🚧 Why it matters

→Product & Architecture never have to ask “what’s the status?” — Jira is the answer.
→Release planning depends on accurate Jira state at Task & Sub-task level.
→Auditability: Jira is the long-term source of truth; Beadbox is the local lens.
📣 Ownership: this mirror is the developer’s responsibility. Not the PM’s. Not the architect’s.
