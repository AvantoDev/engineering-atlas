Why it exists

The goal: a fast, consistent first pass

Every PR deserves a thorough review — but human reviewers are busy and inconsistent. The agent gives every PR the same rigorous first look in seconds, so humans spend their time on judgment, not catching the obvious.

🎯 WhatAn autonomous reviewer that reads your diff against the spec + our best-practice rules and posts a real GitHub review.
⚡ WhyConsistency & speed. It never gets tired, never skips the security checklist, never forgets the design-system rules.
🤝 Not a replacementIt's the first reviewer, not the last. A human still approves — the agent just clears the noise first.
Under the hood: a Python service on AWS Bedrock (Claude Sonnet 4.6), running a deterministic LangGraph pipeline so irreversible actions — merge, ticket transitions — always happen in a safe, fixed order.
The first reviewer on every PR — same rigor, every time
How you call it

Tag it in Slack

There's one thing to remember: mention the bot with a PR link. That's it. Post it in the PR thread in #strata-backend-pr and the review starts.

@PR Review Agent review https://github.com/AvantoDev/avanto-db/pull/930
# or just paste a Jira key — it finds the PR:

# or a Slack thread URL · or several PRs at once
It accepts a PR URL, a Jira key (e.g. ST-223), a Slack thread URL, or multiple PRs in one message. No URL handy? It will try to resolve the PR from the ticket for you.

What it looks at

Two layers of checks

Every PR gets the base ruleset. On top of that, the agent detects the repo and adds the repo-specific rules — so a frontend PR gets React & design-system checks a backend PR never sees.

① Base ruleset — every PR
Security / OWASP · injection, authz, secrets
Error handling · failures caught & surfaced
Data integrity · migrations, idempotency
Code quality & testing · coverage of new logic
Spec compliance · does it match the PRD/spec?
② Repo-specific — auto-detected
front-react-strata-ui · React correctness, design-system imports, frontend security, a11y, test integrity
avanto-db · migration & schema rules
graphql-gateway · gateway contract rules
📋 Specs first. Drop a *-spec.md / *-design.md (or anything under **/specs/**) in the PR and the agent reviews against it. Want strict PRD enforcement instead? See the next slide.

How to read the review

Three severities

Findings are grouped so you know what to do at a glance. Only one of them blocks the merge.

🔴 Must Fix
Blocks the merge. Security holes, data-integrity risks, spec violations. The agent will not auto-merge while any of these are open.
🟡 Should Fix
Strongly recommended, non-blocking. Quality, missing tests, risky patterns. Fix now or file a follow-up — your call.
🟢 Informational
FYI. Style notes, suggestions, and anything you ack:'d. Never blocks anything.
The Slack reply leads with Findings: N (X Must Fix, Y Should Fix, Z Informational) so you get the verdict before opening GitHub.
Only 🔴 Must Fix blocks the merge · 🟡 and 🟢 never do
When it merges for you

The auto-merge policy

The agent only merges when it's clearly safe. All five must hold — miss one and it approves but leaves the merge to you.

①The review is an approval — the model is confident.
②Zero 🔴 Must Fix findings open.
③The PR is not a draft.
④The base branch is not protected — main, production, strata-production are never auto-merged.
⑤You didn't say dont merge.
🔒 Protected branches always require a human merge — by design. If anything is unclear, the agent's default is to approve but not merge, never the other way around.
Five green lights or no auto-merge · protected branches always manual
Good to know

What it does in tricky cases

Draft PR
You mention it on a draft.
→ Full COMMENT review with all findings — but no auto-merge, no Jira transition. Mark Ready & re-mention for the full flow.
Merge conflicts
The branch can't merge cleanly.
→ It reviews and tells you; resolve conflicts, then re-mention.
Re-review
You pushed fixes and tag it again.
→ It reconciles against its previous review, so ack:s and resolved items carry over.
Something failed
An API call broke mid-review.
→ Friendly message + correlation_id. Notion failures are non-fatal — the review still posts.
Drafts → comment-only · conflicts surfaced · re-reviews remember · errors are traceable
