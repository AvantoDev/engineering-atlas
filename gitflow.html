The gate

Task → US branch — the gate

Tasks (the tickets inside a US) merge into the US branch via merge-commit PRs; same-US disciplines integrate there — the front Task tests against the back Task. The US branch is a clean delta over production (only its own work).

One US, many repos — same branch name is the coordination key

Testing track

qa accumulates everything — and never ships

In-flight US branches merge into qa so they're tested together (cross-US, front vs back). qa is a sink: merges flow IN, nothing flows out — it is never merged into production.

qa (soup)
●production base MUS ST-1234 MUS ST-1240 MUS ST-1255
$ git checkout qa
$ git merge --no-ff origin/ST-1234  # merge US branches IN to test together
$ git push origin qa  # deploys the soup to the qa env — NOT a release
qa is a sink · never merged into production · never commit directly to it
Release track

Assemble a release · ship via pre-main

A disposable release/<cycle> branch is cut from production; only the ready US branches are volcadas in. It's promoted to production, where the pre-main (k8s) gate validates only the release before go-live.

$ git checkout production && git pull
$ git checkout -b release/2026-06  # disposable, fresh off production
$ git merge --no-ff origin/ST-1234  # volcar ONLY ready US branches
$ git merge --no-ff origin/ST-1240
$ # merge release/2026-06 → production → pre-main (k8s) validates only the release → approve → live 🚀
📦 The release is assembled from production + the ready US branches — never from qa.
release branch is disposable · rebuilt per cycle · qa is not the source
Two tracks, no leaks

Why qa never contaminates production

qa is a sink, not a source: you merge US → qa, never the reverse. Each US stays a clean delta over production, so when it ships via the release branch it brings nothing from the soup.

1US & release branches are cut from production — never from qa.
2qa is a sink — merge US → qa, but never merge qa back into a US / release / production branch.
3The release branch is built from production + ready US branches — never from qa.
4Sync = git merge production into a US branch — never qa.
5Integration fixes go to the US/Task branch (then re-merge to qa) — never direct commits to qa.
🔴 Inverse gotcha: a fix committed directly to qa (not the US branch) lives in qa but is missing from production when the US ships.
The classic traps

…and the one move that beats each

① Drop a US late
It's in qa and queued for the release.
→ Rebuild the release branch without it. qa untouched.
② Stale US vs moved prod
production advanced; the US branch is behind.
→ git merge production into the US branch, then re-validate.
③ Urgent task in a half-done US
One piece is needed now, the rest isn't.
→ Fix-forward off production as its own PR.
④ Code shared by two US's
Both need the same change.
→ Land it first to production; both pick it up via sync.
None of these is ever solved with a cherry-pick
Undo = rebuild / revert

Drop a US · roll back a release

production
● ● MRelease 2026-06 Rrevert -m 1
Drop one US
Rebuild the release branch = production + the remaining ready US branches. qa untouched (the US keeps testing there).
Roll back the whole release
git revert -m 1 <release merge> + redeploy. A new commit undoes it — no history rewrite.
🏳️ Without flags, the "off switch" is a rebuild or a revert + redeploy — slower than a flag flip, but the team already knows it.
Drift & sync

Keep US branches current — merge, not rebase

T0production = A. Cut US ST-1234 (off A); it touches classifier.py.
T1Hotfix H fixes a bug in classifier.py → production: A → B. ST-1234 doesn't have H — shipping it now would re-introduce the bug. 🔴
T2Sync: git merge production into ST-1234 → now = B + my changes, includes H, conflicts resolved here. ✅
❌ Don't rebase the US branch — it's shared (Task PRs target it); rebase force-pushes and breaks them. 🟢 Cheapest defense: small, short-lived US's.
Sync after every release and every hotfix
Definition of Ready · ownership

Developer prepares · Lead is a gate

A US is release-ready only when, in every repo it touches, its developer confirms: ① self-contained · ② synced (git merge production) · ③ integrated QA passed (validated in qa) · ④ PR(s) green + ready-to-release.

Responsibility
Owner
Keep the US release-ready (the 4 points), per repo
Developer
git merge production + resolve conflicts
Developer
Verify readiness before a US is volcada into the release branch
Eng Lead (gate)
Pick which ready US's ship — in any order
Product + Arch
🎯 The Lead gates entry to the release branch, not to qa (US branches merge to qa freely to test). Ready US's wait in a pool; a release picks any subset, any order.

The rules · JH-06 … JH-13

Two tracks. Cherry-pick-free.

JH-06 Task → US branch (the gate); US = release unit; merge-commit, never rebase-merge
JH-07 qa = integration accumulator · a sink · never promoted
JH-08 disposable release branch → production (pre-main k8s gate)
JH-09 no cherry-pick: drop = rebuild release branch; rollback = revert
JH-10 guardrails keep qa out of production (qa is a sink)
JH-11 fix-forward; hotfix off production
JH-12 sync via merge production (not rebase); small US's
JH-13 Definition of Ready is developer-owned; Lead is the gate
