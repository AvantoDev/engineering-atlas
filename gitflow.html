The big idea

Day-by-day tutorial

Let's follow one Story: ST-1234

A real example, day by day — so you can repeat it on your own work. Five moments, start to ship:

Day 1Start. Find or create the shared US branch; branch your Task off it.
Day 2Test. Merge the US branch into qa and test it with everyone else's.
Day 3Sync. A release shipped — pull production into the US branch.
Day 4Ready. Tests pass — mark it ready-to-release. You're done.
ShipRelease day. The Lead ships your Story with the release.
Tutorial · follow ST-1234 from Day 1 to release
Day 1 Start your work

Find or create the Story branch

You pick up a Task in Story ST-1234. First, the US branch (ST-1234) — it's shared by the whole Story team. Does it exist? If yes, reuse it; if a teammate hasn't made it yet, you create it from production. Then branch your Task off it.

$ git fetch origin
$ git switch ST-1234  # reuse it if it already exists…
$ # …or create it ONLY if it doesn't: git switch -c ST-1234 origin/production
$ git switch -c ST-1235-backend  # YOUR Task branch, off the US branch
$ gh pr create --base ST-1234  # PR your Task INTO the shared US branch (merge-commit)
The US branch is the Story's (shared) — never create a second one if it exists. Your Task branch is yours. Its readiness is owned by the mentor (Rey Rivero / Faizan Pervaz / Ateeb Tasser), then the Tech Lead.
Day 1 · reuse ST-1234 (or create from production) · branch your Task off it
Day 2 Test it in qa

Merge into qa to test together

Your Task is merged into the US branch. Now test the Story integrated with everyone else's work — merge the US branch into qa; it deploys to the QA environment (where, e.g., the frontend runs against the backend already live in qa).

$ git checkout qa && git pull
$ git merge --no-ff ST-1234  # bring the US branch IN to test
$ git push origin qa  # deploys to the QA environment
⚠️ qa is the shared test environment — it never goes to production. Found a bug? Fix it on your US branch and merge again. Never commit a fix directly to qa (it would be in qa but missing from production).
Day 2 · merge the US branch → qa · test · qa never ships
Day 3 Stay in sync

A release shipped — pull it in

While you were working, a release (or a hotfix) went to production. It moved forward. Pull it into the US branch so it doesn't drift — and so you don't accidentally re-introduce a bug a hotfix already fixed.

$ git checkout ST-1234
$ git merge production  # MERGE — never rebase
🟢 Always merge production into the US branch — never rebase it (it's shared; Task PRs point at it, so a rebase force-pushes and breaks them). Any dev on the Story can run this; the mentor is accountable for the US branch staying in sync. Do it after every release or hotfix.
Day 3 · git merge production into the US branch · merge, not rebase
Day 4 Mark it ready

Ready means ready — then you're done

The Story is release-ready when these four hold. The mentor (Rey Rivero / Faizan Pervaz / Ateeb Tasser) owns the US branch's readiness — sync, conflicts, marking it ready; the Tech Lead is the final escalation:

①Self-contained — only this Story's work; no dependency on an unreleased Story.
②Synced — git merge production is up to date.
③Tested — validated integrated in qa.
④Green — PRs pass + label ready-to-release.
🎯 You own your Task. The mentor owns the Story branch and hands it to the release — you don't assemble or promote. If the Story isn't ready, it simply waits for the next release.
Day 4 · all four green · the mentor marks it ready & hands it to the release

The rules · keep these handy

✅ Always · ❌ Never

✅ Always
  • Reuse the Story's US branch — create from production only if missing
  • PR your Task into the US branch (merge-commit)
  • Merge the US branch → qa only to test
  • Sync the US branch with git merge production
  • Fix bugs on the branch, then merge again
❌ Never
  • Push to production, or cut a branch from qa
  • Create a second US branch if one already exists
  • Merge qa into a US / release / production branch
  • Commit a fix directly to qa
  • Rebase the US branch · cherry-pick anything
production is the only source for branches · qa is test-only · no cherry-pick
When things go sideways

If something goes wrong

Bug in your Story
Found while testing in qa.
→ Add more commits on your US branch (fix-forward), then merge to qa again. Don't delete the branch.
Urgent production fix
Something is broken in production now.
→ Hotfix: branch off production → PR back to production. Then everyone syncs.
Your Story must be pulled from a release
It's not ready in time.
→ The Lead rebuilds the release without it. You never cherry-pick. Your branch waits in qa.
production moved ahead of you
A release shipped mid-work.
→ git merge production into the US branch (Day 3). Re-test.
Every fix is forward · nothing is ever cherry-picked
