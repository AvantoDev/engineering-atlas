One Story, many repos

The branch name is the coordination key

A Story crosses repos (AR-01). Each repo gets its own ST-1234 branch — same name everywhere.

🔑 Discoverable by name: git branch -r | grep ST-1234 finds every piece of the Story, in any repo. That is what makes cherry-picking unnecessary.
Release = promote the ST-1234 set across all its repos
The testing track

qa accumulates everything — and never ships

In-flight US branches merge into qa so they're tested together (cross-US, front vs back). qa is a sink: merges flow IN, nothing flows out — it is never merged into production.

qa (soup)
●production base MUS ST-1234 MUS ST-1240 MUS ST-1255
$ git checkout qa
$ git merge --no-ff origin/ST-1234  # merge US branches IN to test
$ git push origin qa  # deploys the soup to the qa env — NOT a release
qa is a sink · never merged into production · never commit directly to it
The two ground truths

The Story is the release unit.
There is no cherry-pick.

Every "undo" in this flow is a revert — of a PR, or of the release merge. Drops, rollbacks, hotfixes: all branch-native, all reversible, all with tools the team already uses.

The four classic traps

…and the one move that beats each

① Drop a US late
It's in qa and queued for the release.
→ Rebuild the release branch without it. qa untouched.
② Stale Story vs moved prod
production advanced; the branch is behind.
→ git merge production before it ships. Re-validate.
③ Urgent task in a half-Story
One piece is needed now, the rest isn't.
→ Fix-forward off production as its own PR.
④ Code shared by two Stories
Both need the same change.
→ Land it first to production; both merge it in.
None of these is ever solved with a cherry-pick
Undo = revert

Drop a US · roll back a release

production
● ● MRelease 2026-06 Rrevert -m 1
Drop one US
Rebuild the release branch = production + the remaining ready US branches. qa untouched (the US keeps testing there).
Roll back the whole release
git revert -m 1 <release merge> + redeploy. A new commit undoes it — no history rewrite.
🏳️ Without flags, the "off switch" is a revert + redeploy — slower than a flag flip, but the team already knows it.
Drift & sync

Keep Stories current — merge, not rebase

T0production = A. Cut ST-1234 (off A); it touches classifier.py.
T1Hotfix H fixes a bug in classifier.py → production: A → B. ST-1234 doesn't have H — merging now would re-introduce the bug. 🔴
T2Sync: git merge production into ST-1234 → now = B + my changes, includes H, conflicts resolved here. ✅
❌ Don't rebase the Story branch — it's shared (Task PRs target it); rebase force-pushes and breaks them. 🟢 Cheapest defense: small, short-lived Stories.
Sync after every release and every hotfix
Definition of Ready · ownership

Developer prepares · Lead is a gate

A Story is release-ready only when, in every repo it touches, its developer confirms: ① self-contained · ② synced · ③ isolated QA passed · ④ PR green + ready-to-release.

Responsibility
Owner
Keep the Story release-ready (the 4 points), per repo
Developer
git merge production + resolve conflicts
Developer
Verify the Definition of Ready before qa entry
Eng Lead (gate)
Pick which ready Stories ship — in any order
Product + Arch
🎯 Out-of-order is fine: ready Stories wait in a pool; a release picks any subset, any order. The Story is the sum of its per-repo ticks — no central preparer.

The rules · JH-06 … JH-13

Two tracks. Cherry-pick-free.

JH-06 Task → US branch (the gate); US = release unit; merge-commit, never rebase-merge
JH-07 qa = integration accumulator · a sink · never promoted
JH-08 disposable release branch → production (pre-main k8s gate)
JH-09 no cherry-pick: drop = rebuild release branch; rollback = revert
JH-10 guardrails keep qa out of production (qa is a sink)
JH-11 fix-forward; hotfix off production
JH-12 sync via merge production (not rebase); small US's
JH-13 Definition of Ready is developer-owned; Lead is the gate

🏳️ Feature flags layer on top later — without changing any of this.
